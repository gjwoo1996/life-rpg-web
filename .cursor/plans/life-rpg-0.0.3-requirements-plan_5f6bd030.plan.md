### 목표

- **0.0.3 개선 요구사항 문서를 기준으로 앞으로 구현해야 할 기능/수정 항목을 구조화**하고, 
- **각 항목을 코드/문서 차원에서 어디를 어떻게 손댈지 초안 수준의 계획**을 세운다.

### 1. 캐릭터 기능 개선 계획

- **Phaser.js 도입 및 도트 캐릭터 적용 전략 정리**
  - 현재 프론트엔드 스택(React/Next 등)을 파악한 뒤, Phaser.js를 어느 계층에서 붙일지 결정 (별도 캔버스 컴포넌트 vs 독립 페이지 등).
  - 캐릭터 렌더링을 담당할 `CharacterScene`(가칭) 및 React 래퍼 컴포넌트 설계 초안 수립.
- **레벨 단계 세분화 및 스프라이트 설계 방향**
  - 기존 캐릭터 레벨 구조(예: 1, 5, 10 단위 변화 등)를 확인하고, **1레벨 단위로 스킨/프레임을 바꿀 수 있는지** 데이터 구조 관점에서 검토.
  - 캐릭터 상태를 표현하는 데이터 모델(예: `level`, `stage`, `appearanceId` 등)을 정의하고, 레벨 변화 → 스프라이트 매핑 규칙을 문서화.
- **캐릭터 성장 테스트 페이지 계획**
  - `/character-test`(가칭) 단일 페이지를 만들고, 해당 페이지에서 **레벨 슬라이더/버튼을 통해 캐릭터 상태를 즉시 바꿔보는 데모 UI**를 설계.
  - 테스트 페이지에서 사용할 임시 API 또는 mock 데이터 구조를 정의하고, 향후 실제 게임 데이터와 어떻게 연동할지 방향만 잡아둔다.
- **도트 캐릭터 리소스/질문 정리**
  - 도트 캐릭터 구현에 필요한 정보(프레임 수, 방향, 애니메이션 패턴, 해상도, 아트 리소스 포맷 등)를 체크리스트로 정리.
  - 이 체크리스트를 기반으로, 추후 질문해야 할 항목 목록을 문서에 기록해 둘 계획을 세운다.

### 2. 불필요 기능 개선 및 삭제 계획

- **멀티 캐릭터 가정 제거 범위 파악**
  - 라우팅/메뉴/상태 관리에서 **캐릭터 선택을 전제로 한 UI/로직**을 탐색할 위치를 정리 (예: `CharacterSelect` 컴포넌트, 캐릭터 ID 배열을 다루는 스토어 등).
  - 캘린더 진입 전 캐릭터 선택을 요구하는 부분처럼, 대표적인 플로우를 2~3개 예시로 골라 우선순위화.
- **싱글 캐릭터 모델로의 단순화 방향**
  - 전역 상태(예: `currentCharacterId`)를 단일 값으로 고정하고, 배열/리스트 구조를 제거하는 리팩터링 전략 수립.
  - URL 파라미터나 쿼리에서 캐릭터 ID를 받는 패턴이 있다면, 이를 제거하거나 기본 캐릭터로 강제 매핑하는 규칙을 문서화.

### 3. 목표 기능(Goal/Objective) 개선 계획

- **목표 데이터 모델 변경 설계**
  - 기존 목표 스키마(예: `name`, `startDate`, `endDate`, `abilityId` 등)를 `title`, `description`, `startDate`, `endDate` 중심으로 재정의하는 초안 작성.
  - DB/백엔드가 있다면 마이그레이션 전략(필드 이름 변경, 의미 변경)을 개략적으로 정리.
- **스텝(step) 구조 및 별도 페이지 설계**
  - 목표 하나에 여러 스텝을 가지는 구조를 정의 (예: `steps: [{ id, title, description, date }, ...]`).
  - `Step 설정 페이지`(예: `/goals/:goalId/steps`)의 화면 흐름을 정의: 스텝 추가/수정/삭제, 날짜 선택, 내용 입력.
- **분석 기능 수정 방향**
  - 기존 분석 기능이 `연결할 능력`을 기준으로 통계를 내고 있었다면, 새 구조에서 **스텝 단위, 목표 단위, 기간 단위**로 어떻게 재해석할지 요약.
  - 분석 화면/그래프에서 어떤 항목을 유지·변경·삭제할지 목록화.

### 4. 메뉴 추가(성장그래프) 계획

- **성장그래프 메뉴 구조 설계**
  - 상단/사이드 메뉴에 `성장그래프` 항목을 어디에 배치할지(예: `리포트` 섹션 하위) 정의.
  - 해당 메뉴 클릭 시 이동할 경로(예: `/growth` 또는 `/reports/growth`)를 정한다.
- **그래프 타입 및 데이터 정의**
  - 일일/주간/월간 성장에서 각각 어떤 지표를 사용할지(경험치, 레벨, 완료한 목표/스텝 수 등) 초안 정의.
  - 차트 라이브러리(예: Chart.js, Recharts 등) 후보를 정하고, 각 기간별로 공통 컴포넌트로 구현할지 여부를 계획.

### 5. Cursor 룰 수정 계획

- **eslint 및 any 사용 규칙 반영**
  - `.cursor/rules/` 내 ESLint 관련 룰/코딩 규칙 문서를 확인하고, `any` 사용 자제 및 ESLint 통과를 명시하는 섹션을 추가하는 방향을 잡는다.
  - 향후 타입 설계 시 `any` 대신 사용할 수 있는 패턴(예: 제네릭, 유니온 타입, 구체 인터페이스 등)을 간단히 언급.

### 6. 문서 업데이트 연동 계획

- **docs 업데이트 규칙 반영**
  - `docs/개선요구사항/0.0.3.md`를 기반으로 실제 구현에 들어갈 때, 완료된 항목을 `docs/업데이트/`의 버전별 변경 내역 문서로 옮겨갈 계획을 세운다.
  - 예: 0.0.3 요구사항 중 일부를 구현한 경우, `docs/업데이트/0.0.3.md`(또는 다음 버전)에서 "캐릭터 성장 테스트 페이지 추가", "목표 스텝 기능 도입" 등으로 요약.
